# R — Reader

## Purpose

R exists for one purpose:

To read Scripture.

Nothing should compete with the biblical text.

The Reader is intentionally quiet.

The software should disappear so the text can remain central.

---

## Philosophy

The biblical text is the interface.

Not the menus.

Not the toolbars.

Not the application.

The Reader exists to serve the text.

---

## Design Principles

### The text occupies the screen.

Whenever possible, the text should be the largest and most prominent element.

---

### Menus disappear.

Controls should remain hidden until needed.

The reader should spend time looking at Scripture, not the interface.

---

### Every feature begins with the text.

No feature should require leaving the reading experience.

Observation happens inside the text.

Study happens inside the text.

Navigation happens inside the text.

---

### Reading comes first.

Every additional capability must preserve the reading experience.

If a feature interrupts reading, it should be redesigned or removed.

---

## Relationship to Other Modules

R is the foundation.

Other modules assist reading.

They should never replace it.

Observation should emerge from reading.

Translation should emerge from reading.

Study should emerge from reading.

The Reader remains the center.

## Silence

Good reading software is quiet.

The Reader should rarely ask the user to make decisions about the software.

Instead, it should invite the user to make observations about the text.

If the interface becomes more interesting than Scripture, the design has failed.

## The Reader is not a destination.

The Reader is a doorway.

Every interaction should return the reader's attention to the biblical text as quickly as possible.

The Reader should never become the object of attention.

Scripture is the object.

## Tools

The Reader provides tools, not features.

A tool helps the reader observe one aspect of the biblical text.

When its purpose is complete, it quietly steps aside.

The Reader should never become more interesting than the text it presents.

---

## Every Tool Answers One Question

Every tool exists to answer one observable question.

Examples

Finite Verbs
→ Where are the finite verbs?

Commands
→ Where are the commands?

Command Recipients
→ Who receives the commands?

Dependent Clause Introducers
→ Which words introduce dependent thoughts?

Clause Builder
→ What words belong to this finite verb?

Skeleton
→ What remains when dependent thoughts are hidden?

If a tool begins answering multiple questions, it should probably become multiple tools.

---

## Quiet Design

The best reading software is almost invisible.

The biblical text is the interface.

Menus should remain minimal.

Controls should appear only when needed.

The reader's attention should continually return to Scripture.

Every design decision should ask:

Does this help the reader observe the text?

Or does it draw attention to the software?

If it draws attention to the software, it should be redesigned or removed.
